1. מבוא

מסמך זה מתאר את התכן ההנדסי של מערכת Kali Tools GUI: ממשק גרפי ושכבת סוכנים בגישת Purple Team המורצת מעל כלי Kali Linux תחת WSL2. המסמך נועד להסביר כיצד המערכת בנויה, כיצד רכיביה משתלבים, ובמיוחד — מאיפה הסוכנים "יודעים" לפעול ולקבל נתונים.

איור 1 — שכבות המערכת וזרימת הבקרה.

3. שכבת ה‑Backend (server.py)

שרת HTTP רב‑תהליכי (ThreadingHTTPServer) בספריית פייתון הסטנדרטית. מגיש את קבצי ה‑Frontend וחושף API ב‑JSON.

3.1 מודל ה‑Job

כל הרצת כלי היא אובייקט Job: מריץ subprocess.Popen בקבוצת תהליכים נפרדת, קורא פלט בתהליכון (thread) עם מגבלת גודל, ומאפשר עצירה נקייה. הפלט זמין ל‑polling מהדפדפן.

3.2 נקודות קצה עיקריות

נקודת קצהתפקיד
GET /api/toolsקטלוג הכלים + סטטוס התקנה
POST /api/runהרצת כלי בודד
POST /api/planPlanner: כוונה+מטרה → תוכנית שלבים

4. מנוע הכלים ובניית פקודות

הקטלוג tools.json מגדיר לכל כלי: בינארי, קטגוריה, ואת שדות הטופס (סוג, דגל, ברירת מחדל). פונקציית build_argv בונה את הפקודה כמערך argv — כל אסימון בנפרד, ללא shell. זהו עיקרון אבטחה מרכזי: ערכי המשתמש לעולם אינם מתפרשים ע"י מעטפת, ולכן הזרקת פקודות נמנעת.

argv = [binary]
for opt in options:            # לפי סדר ההגדרה
    if bool ומסומן:  argv += [flag]
    elif ערך קיים:   argv += [flag, value]   # או "flag=value"
argv += [positional target]

5. שכבת הסוכנים (agents.py)

מנוע מבוסס‑חוקים (Playbooks) — ללא תלות ב‑LLM, דטרמיניסטי וניתן לביקורת.

🧭 Planner (מתכנן)
מקבל כוונה חופשית + מטרה. מנקד כל Playbook לפי מילות מפתח (עברית+אנגלית), בוחר את המתאים ובונה שלבים (כלי + פקודה + הסבר + הצעה). לדוגמה: "בדיקת DNS" → dig/dnsenum/dnsrecon.
▶️ Executor (מבצע)
מריץ כל שלב ברצף כ‑Job, עם מגבלת זמן לכל שלב (ברירת מחדל 300 שניות) כדי שכלי איטי לא יתקע את המשימה.
🔎 Verifier (מאמת)
מנתח את הפלט וקוד היציאה של הכלי. באמצעות דפוסי regex מזהה שגיאות (דרוש root, מטרה לא זמינה) ומחלץ ממצאים (פורטים פתוחים, CVE, חולשות). מפיק verdict: תקין / ממצאים / אזהרה / נכשל.
📄 Reporter (מדווח)
מרכיב דוח Markdown: סיכום מנהלים, ממצאים עיקריים ופירוט שלבים. אם מוגדר Ollama — משדרג את הניסוח ב‑LLM.

6. שכבת Purple Team (bluered.py)

🟢 Broker (מתווך)
מקבל את ממצאי הצוות האדום, ממפה כל ממצא לכלל הגנה מתוך DEFENSE_KB, מאחד עדויות ומדרג לפי חומרה.
🔵 Blue Team (צוות כחול)
בסיס ידע של ~15 כללי הגנה. לכל איום: תיאור, פעולות הגנה/הקשחה, זיהוי וניטור, תצורה לדוגמה, ומיפוי MITRE ATT&CK (למשל T1190, T1110).
🧠 Learning (למידה)
כל ממצא מנורמל לחתימה ונצבר ב‑knowledge.json (מונה + first/last seen). כך המערכת "לומדת" אילו איומים חוזרים לאורך זמן.
🟣 Orchestrator (מתזמר)
מנצח את כל הזרימה: מריץ את הצוות האדום, מפעיל את המתווך והכחול, מעדכן את הלמידה, ומפיק דוח Purple מלא.
🔧 Remediation (סוכן מתקן)
ממפה כל חתימת איום לתיקון מעשי (למשל fail2ban, עדכוני אבטחה) ומיישם אותו על המערכת — רק לאחר אישור מפורש, עם גיבוי קונפיגים ורישום ביומן ביקורת. תיקונים מסוכנים (SSH/firewall) מסומנים כ"ידני" ואינם רצים אוטומטית, למניעת נעילה.

7. זרימת נתונים — מאיפה הסוכנים יודעים?

שאלת מפתח: הסוכנים אינם "ממציאים" נתונים. הם שואבים משלושה מקורות ברורים וניתנים לביקורת.
מקורמה הוא מכילמי משתמש בו
1. ידע סטטי (קוד/JSON) tools.json (קטלוג ופקודות), agents.py (Playbooks + דפוסי חילוץ), bluered.py (כללי הגנה + MITRE) Planner, Verifier, Blue
2. נתונים אמיתיים (הכלים) הפלט האמיתי של כלי Kali על המטרה — פורטים, גרסאות, חולשות. אמת הקרקע. Executor → Verifier
3. זיכרון מצטבר (קבצים) knowledge.json (למידה), activity.json (יומן), reports/ (דוחות) Learning, Dashboard, Brain
intent + target │ ▼ [Planner] ── reads tools.json + PLAYBOOKS ──► plan (steps) │ ▼ [Executor] ── subprocess ──► REAL Kali tool ──► raw output │ │ ▼ ▼ [Verifier] ── regex patterns ──────────────► findings │ ▼ [Broker] ── maps to DEFENSE_KB ──► threats + defenses │ ▼ [Learning] ── writes ──► knowledge.json (grows every run) │ ▼ [Reporter] ──► report.md ──► reports/ + activity.json

איור 2 — זרימת הנתונים בין הסוכנים. הממצאים האמיתיים מגיעים תמיד מהכלים עצמם.

8. שכבת ה‑Frontend ותצוגת המוח

אפליקציית עמוד יחיד (SPA) ב‑vanilla JS ללא שלב build. שלושה מצבים: כלים, עוזר AI, ומרכז בקרה. הדשבורד מציג את מצב הסוכנים החי (נגזר מהרצות פעילות), מודיעין נצבר ויומן פעילות.

8.1 תצוגת "מוח סוכן"

לחיצה על סוכן פותחת ויזואליזציה עתידנית מבוססת SVG: ליבה זוהרת במרכז, יכולות במסלולים סביבה, סינפסות עם חלקיקי אנרגיה, וליבה "נושמת". המנגנון המרכזי הוא Semantic Zoom (Level-of-Detail): ככל שמתקרבים (גלגלת) נחשפות שכבות עמוקות יותר — יכולות → כלים → פרטים. הנתונים נשאבים מ‑brain.json ומועשרים בנתונים חיים (כלים מותקנים, ממצאים ידועים) דרך /api/brain.

9. אבטחה

  • בניית פקודות בטוחה: מערך argv ללא shell — מונע הזרקת פקודות.
  • האזנה מקומית: ברירת מחדל 127.0.0.1 בלבד — התהליך (root) אינו חשוף לרשת.
  • אימות אופציונלי: טוקן משותף (KALIGUI_TOKEN) כהגנה נוספת.
  • מניעת traversal: הגשת קבצים סטטיים מוגבלת לתיקיית web בלבד.
  • שלב אישור: שכבת ה‑AI מציעה תוכנית; ההרצה (וכל תיקון) מתבצעת רק לאחר אישור המשתמש.
  • אימות בענן: התחברות Google + רשימת מיילים מאושרים (oauth2‑proxy) לגישה מבחוץ.
  • ביקורת: כל פעולה נרשמת ב‑audit.log עם זהות המשתמש שביצע אותה.

10. פריסה ותפעול בייצור

המערכת מותקנת כשירות systemd בתוך Kali WSL (kali-gui.service) עם הפעלה אוטומטית והתאוששות מקריסה (Restart=on-failure). מתקין חד‑פעמי (Install.bat / setup.ps1) מבצע: בדיקת דרישות, תיקון DNS, התקנת השירות, ורישום משימת Windows להעלאת WSL בהתחברות.

systemctl status  kali-gui     # מצב
systemctl restart kali-gui     # הפעלה מחדש
journalctl -u kali-gui -n 50   # לוגים

עמידות: טיפול גלובלי בשגיאות בכל בקשה, גיזום היסטוריית הרצות בזיכרון, לוגים לקובץ ול‑journald, ושמירת דוחות לדיסק (שורדים אתחול).

10.1 פריסה בענן מאובטחת (VPS)

לגישה מבחוץ, האפליקציה נשארת על 127.0.0.1 ומעליה שרשרת הגנה:

Internet ──HTTPS(443)──► Caddy ──► oauth2-proxy ──► app (127.0.0.1:8777) TLS auto (Let's Encrypt) Google login + email allowlist

ה‑oauth2‑proxy אוכף התחברות Google ומאמת מול רשימת מיילים מאושרים (האדמין מוסיף/מסיר משתמשים בקובץ). המשתמש המאומת מועבר לאפליקציה בכותרת X‑Forwarded‑Email. לוג ביקורת תלת‑שכבתי: Caddy (גישות) · oauth2‑proxy (התחברויות) · האפליקציה (audit.log — מי הריץ/תיקן מה). מתקין אוטומטי (install-vps.sh) מקים את הכל כולל firewall (ufw) ו‑fail2ban, וקיימת גם חלופת docker‑compose.

10.2 שרת MCP

רכיב mcp/kali_gui_mcp.py חושף את ה‑REST API ככלי MCP — כך ש‑Claude (או כל לקוח MCP) יכול להפעיל את המערכת בשיחה: להריץ כלים, לתכנן ולהריץ בדיקות ו‑Purple, לקרוא מודיעין, ולהחיל תיקונים (עם אישור). הוא client דק מעל ה‑API; הליבה נשארת ללא תלויות.
