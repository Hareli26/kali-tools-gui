3. קשרים והחלטות תכן

קשרסוגמשמעות
hp_pots.id ← hp_events.pot1 : Nכל מלכודת כותבת אירועים רבים
hp_signatures.sig ← hp_events.technique1 : Nכל אירוע מסווג לטכניקת תקיפה
hp_geo.ip ← hp_events.src_ip1 : Nכל כתובת תוקף מועשרת פעם אחת (JOIN בתצוגות)
hp_signatures ⟷ signatures🔥 הצלבההלב של המערכת: hp_correlate() מצליב "מה תוקפים מנסים" מול "מה חלש אצלנו" — קשר לוגי במפה בקוד, לא ב-DB
למה שני בסיסי חתימות נפרדים (signatures ו-hp_signatures)? הם טוענים טענות הפוכות: signatures = "למטרה יש חולשה" (תנוחה, מהצוות האדום שלנו); hp_signatures = "מישהו ניסה את זה" (התנהגות, מהמלכודות — קלט של תוקף). ערבוב היה מזהם את הסטטיסטיקות ומאפשר לתוקף לכתוב לבסיס הידע שלנו על עצמנו. לכן טבלאות נפרדות — וההצלבה ביניהן היא הערך.

3.1 עקרונות תכן

  • SQLite, ספרייה סטנדרטית (db.py) — אפס תלויות. הגירה אוטומטית חד-פעמית מקבצי JSON ישנים (knowledge.json, activity.json) ו-ALTER TABLE להוספת עמודות ה-OSINT ל-hp_geo קיים.
  • ללא אילוצי FK — הקשרים לוגיים ונאכפים בקוד (JOIN-ים ב-db.py). מתאים למאגר תיעודי קליל שבו חלק מהשדות הם JSON (data, blob).
  • גישה בטוחה לזרימות — כל פעולה עטופה ב-RLock גלובלי + חיבור עם timeout; rollback journal (נייד בין מערכות קבצים).
  • גיזום — hp_events ל-20K שורות, activity ל-500 — כדי שהמאגר לא יגדל ללא גבול תחת מלכודת עמוסה.
  • נתוני תוקף = קלט לא מהימן — hp_events/hp_signatures/hp_geo אף פעם לא מעורבבים עם signatures, ולעולם לא מפעילים תיקון אוטומטי.
